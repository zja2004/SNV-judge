## Slide 3 · 研究背景与意义（约 60 秒）

先说临床痛点。

随着二代测序技术的普及，全外显子组测序（WES）已经成为遗传病诊断的常规手段。但每做一次 WES，就会产生大约 **2 万个单核苷酸变异（SNV）**。这 2 万个变异里，绝大多数是良性的多态性，真正有临床意义的致病变异可能只有几个甚至几十个。如何从这 2 万个变异里快速、准确地找出那几个"罪犯"，是临床遗传学面临的核心挑战。

现有的主流工具，比如 SIFT、PolyPhen-2，都是基于单一维度——要么是序列保守性，要么是蛋白质结构——来做预测的，AUROC 大约在 0.85 左右，还有相当大的提升空间。

与此同时，近两年出现了两个非常重要的基因组基础模型：Arc Institute 和 NVIDIA 联合发布的 **Evo2**，参数量 400 亿，在 9.3 万亿个核苷酸 token 上预训练，能够捕捉跨物种的进化约束信息；以及浙江实验室发布的 **Genos**，专门针对人类基因组设计，直接输出致病性评分。这两个模型蕴含了极其丰富的进化和功能信息，但目前还没有人系统地把它们整合进变异致病性预测的集成框架里。

这就是我们这个项目的出发点。

## Slide 4 · 数据集与特征工程（约 60 秒）

数据来源是 **ClinVar**，这是 NCBI 维护的人类变异临床意义数据库，我们筛选了经过专家委员会审核（2星及以上）的错义 SNV，最终得到致病变异和良性变异各 **4,723 条**，共 9,446 条，覆盖 547 个基因。数据集按 7:1.5:1.5 划分训练、验证和测试集。

特征矩阵一共 **8 个维度**，分四类：

第一类是**经典工具**：SIFT（氨基酸替换耐受性）、PolyPhen-2（结构功能影响）、AlphaMissense（Google DeepMind 的蛋白质功能预测）、CADD Phred（综合有害性评分）。这四个特征通过 Ensembl VEP REST API 批量获取。

第二类是**基础模型特征**：Evo2-40B 的对数似然比（LLR）和 Genos-10B 的致病性评分。LLR 的计算逻辑是：以变异位点为中心取 101bp 的基因组窗口，分别计算参考等位基因和替代等位基因在 Evo2 模型下的对数概率，两者之差就是 LLR。负值越大，说明替代等位基因在进化上越不被允许，致病可能性越高。

第三类是**保守性**：phyloP 分数，衡量跨物种进化保守程度，通过 Ensembl VEP 的 Conservation 注释获取。

第四类是**人群频率**：gnomAD v4 的等位基因频率，通过 gnomAD GraphQL API 查询。这个特征对应 ACMG 分类标准里的 BA1（常见变异，良性强证据）和 PM2（极罕见，致病中等证据）规则。

## Slide 5 · 模型架构（约 60 秒）

【指向架构图】

整个系统的架构分五个层次，从左到右依次是：

**第一层，输入层**：接收变异的基因组坐标（染色体、位置、参考/替代等位基因）。

**第二层，工具调用层**：并行调用四个 API——Ensembl VEP、NVIDIA NIM（Evo2）、Stomics Cloud（Genos）、gnomAD GraphQL——获取原始分数。

**第三层，特征矩阵**：将所有原始分数整合成 8 维特征向量，缺失值用训练集中位数填充。

**第四层，集成模型**：这是预测的核心。我们用 XGBoost 和 LightGBM 作为基学习器，用逻辑回归作为元学习器，构成一个两层的 Stacking 集成。在此基础上，用 Platt Scaling 做概率校准，使输出的概率值在统计意义上是可靠的。

**第五层，LLM 推理层**：这是方案B新增的部分。SHAP TreeExplainer 计算每个特征对当前预测的贡献值，连同所有工具分数一起，格式化成结构化的 prompt，送入 Kimi 大模型，生成一份 ACMG 风格的临床解读报告。

## Slide 6 · 模型性能（约 60 秒）

【指向 ROC 曲线图】

性能评估采用 5 折交叉验证，并用 1000 次 Bootstrap 重采样计算 95% 置信区间。

核心指标：
- **AUROC = 0.9664**，置信区间 [0.958, 0.972]
- **AUPRC = 0.9671**，置信区间 [0.956, 0.973]
- **F1 = 0.9201**
- **Brier Score = 0.0612**（越低越好，说明概率校准质量高）

从 ROC 曲线可以看到，v4 模型（8 特征）明显优于 v3（7 特征）、v2（6 特征），也优于 AlphaMissense 单工具（AUROC 0.91）和 CADD 单工具（AUROC 0.90）。

这里有一个值得关注的细节：gnomAD AF 这个特征，在有数据的变异上单独 AUROC 可以达到 0.96，但训练集里只有 37% 的变异有非零的 gnomAD AF 值——因为很多 ClinVar 致病变异在人群中极其罕见，gnomAD 根本没有收录。XGBoost 学会了把"gnomAD 里查不到"这个缺失模式本身，也作为一个致病信号来利用，这是树模型相比线性模型的一个重要优势。

## Slide 7 · 消融实验（约 45 秒）

【指向消融图】

为了量化每个特征的独立贡献，我们做了逐特征消融实验——每次移除一个特征，重新训练，观察 AUROC 的变化。

结论非常清晰：

**Evo2-40B LLR 贡献最大**，移除后 AUROC 下降 0.031，这验证了 400 亿参数的 DNA 语言模型确实捕捉到了其他工具无法替代的进化信息。

**Genos-10B 贡献 +0.018**，虽然 Genos 单独的 AUROC 只有 0.65，但它与其他特征存在显著的协同效应，在集成框架里发挥了重要作用。

**phyloP 贡献 +0.012**，**gnomAD AF 贡献 +0.009**。

移除任何一个特征都会导致性能下降，说明 8 个特征各有其不可替代的信息贡献，没有冗余。

## Slide 9 · 方案B：AI 临床报告模块（约 60 秒）

接下来介绍本项目的一个重要创新点——**AI 临床报告模块**，也就是方案B。

这个模块的核心思路是：SNV-judge 给出的是一个数字——比如"致病概率 98.5%"——但临床医生需要的是一份**有理有据的解读**：这个变异为什么被认为是致病的？哪些证据支持？ACMG 分类是什么？

我们用 **Kimi（moonshot-v1-32k）** 作为推理引擎来解决这个问题。Kimi 使用 OpenAI 兼容的 API，支持流式输出，可以在 Streamlit 界面上实时逐字显示报告内容，用户体验非常流畅。

在送入 LLM 之前，我们有一个**证据格式化层**，把原始数值转化为 ACMG 规则语言：
- gnomAD AF 低于 10 的负 7 次方 → 触发 PM2（极罕见，致病中等证据）
- gnomAD AF 高于 0.05 → 触发 BA1（常见变异，良性强证据）
- phyloP 高于 2.0 → 触发 PP3（保守性支持致病）
- Evo2 LLR 低于 -1.0 → 触发 PP3（进化约束支持致病）

这样 LLM 收到的不是原始数字，而是已经映射好的 ACMG 证据条目，大大提高了报告的准确性和一致性。

**左边是 TP53 R175H**，也就是 chr17:7674220 C>T，这是 TP53 基因最常见的热点致病突变，在多种癌症中高频出现。

可以看到，8 个工具全部支持致病：SIFT 评分 1.0（最高有害性）、PolyPhen-2 0.999、AlphaMissense 0.982、CADD Phred 35、Evo2 LLR -2.341（显著负值）、Genos 0.971、phyloP 8.2（高度保守）、gnomAD 未见（AF 极罕见）。Kimi 综合所有证据，给出的分类是**可能致病，概率 98.5%**，ACMG 证据为 PP3×7 加 PM2。

## Slide 13 · 方案C 展望（约 60 秒）

方案B虽然已经实现了 LLM 辅助报告生成，但它本质上还是一个**静态流水线**——工具调用顺序固定，每次只处理一个变异，没有对话能力，也没有记忆。

方案C的目标是把这个系统升级为一个真正的**自主智能体**。

核心区别在于：方案B是"我告诉 LLM 该做什么"，方案C是"LLM 自己决定该做什么"。

具体来说，方案C基于 **LangGraph** 框架，引入一个 **Planner Agent**——由 Kimi K2 驱动——它能够根据用户的自然语言问题，自主规划工具调用顺序，动态决定是否需要查 gnomAD、是否需要调用 Evo2、是否需要检索 ClinVar 文献，然后把多个工具的结果整合起来，给出一个连贯的回答。

同时，方案C引入了**记忆模块**：短期记忆保存当前对话的上下文，长期记忆保存用户的历史报告和偏好，向量数据库存储 ACMG 指南和 ClinVar 文献，支持语义检索。

这样，用户就可以用自然语言和系统对话，比如问"这个变异在东亚人群中的频率是多少"，或者"帮我分析这个 VCF 文件里所有的 TP53 变异"，系统会自主完成所有必要的工具调用，给出完整的回答。

## Slide 14 · 方案C 架构图（约 60 秒）

【指向架构图，从上到下讲解各层】

这张图展示了方案C的完整架构，从上到下分五层：

**最顶层是用户交互层**，用户用自然语言输入问题，比如"chr17:7674220 C>T 这个变异是什么？有临床意义吗？"

**第二层是 Planner Agent**，由 Kimi K2 驱动，基于 LangGraph 的有状态图结构实现。它接收用户问题，自主规划工具调用序列——先调用 VEP 获取基础注释，再调用 gnomAD 查人群频率，再调用 Evo2 和 Genos 获取基础模型评分，然后调用 SNV-judge 集成预测，最后生成 ACMG 报告。整个过程对用户透明，可以实时看到 Agent 的推理步骤。

**第三层是专家工具 Agent 池**，每个工具封装成一个独立的 Agent：VEP Agent、Evo2 Agent、Genos Agent、gnomAD Agent、SNV-judge Agent，以及未来可以扩展的文献检索 Agent。

**第四层是记忆与知识库**，包括短期记忆（对话历史）、长期记忆（用户偏好和历史报告）、向量知识库（ACMG 指南和 ClinVar 文献）、工具调用缓存（避免重复 API 调用）。

**最底层是输出层**，支持结构化 ACMG 报告、多轮对话问答、批量 VCF 分析三种输出模式。

## 附录：常见问题预案

### Q1：为什么选择 Kimi 而不是 GPT-4 或 Claude？
**答**：主要有三个原因。第一，Kimi 是国内合规的大模型服务，在医疗健康场景下数据合规性更有保障；第二，moonshot-v1-32k 提供 32K 上下文窗口，足以容纳完整的变异证据和 SHAP 分析；第三，Kimi 使用 OpenAI 兼容 API，未来可以无缝切换到其他模型（包括 GPT-4、Claude 或本地部署的开源模型），架构上没有锁定。

### Q2：AUROC 0.9664 是在什么数据集上测试的？有没有过拟合风险？
**答**：AUROC 是在严格的 5 折交叉验证下计算的，测试集从未参与训练。我们还用 Bootstrap 重采样计算了 95% 置信区间 [0.958, 0.972]，区间较窄，说明结果稳定。此外，我们控制每个基因最多贡献 10 条变异，避免了少数基因主导训练集导致的过拟合。

### Q3：Evo2 和 Genos 的 API 调用有延迟，实际使用中如何处理？
**答**：我们有两个策略。第一，对于已知变异，我们预计算并缓存了所有 Evo2 和 Genos 分数（存储在 scoring_ckpt.pkl），命中缓存时无需 API 调用，响应时间在 1 秒以内。第二，对于新变异，API 调用约需 2-5 秒，我们在 Streamlit 界面上用进度条提示用户，并支持批量 VCF 模式下的异步并发调用。

### Q4：方案C 的 LangGraph 架构，预计什么时候能实现？
**答**：按照时间规划，方案C的核心框架预计在 2026 年下半年实现。目前方案B已经验证了 LLM 推理层的可行性，方案C是在此基础上的架构升级，主要工作量在于 LangGraph 状态图的设计、工具 Agent 的封装和记忆模块的集成。我们计划先实现一个最小可运行的 demo，再逐步扩展功能。

### Q5：训练数据全部来自 ClinVar，会不会有数据偏差？
**答**：这是一个很好的问题。ClinVar 的变异确实存在基因偏差——BRCA1/2、TP53 等热点基因的变异数量远多于其他基因。我们通过限制每个基因最多 10 条变异来缓解这个问题，但偏差不可能完全消除。在后续工作中，我们计划引入 gnomAD 的人群变异数据作为补充，并在独立的临床验证数据集（如 ClinGen 专家委员会数据）上评估泛化性能。
